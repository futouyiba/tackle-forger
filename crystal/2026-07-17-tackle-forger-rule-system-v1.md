# 结晶：钓具配置工坊——装备生成与规则执行系统设计 v1

## 已确认结论

### 1. 产品目标与边界

钓具配置工坊是面向淡水路亚游戏装备的内部配置工作台。它不是单件装备编辑器，而是一条“定义基准 → 组合生成 → 规则计算 → 人工审阅 → 正式发布 → 从精调中学习”的生产链。

当前只处理三种道具：鱼竿、渔轮、鱼线。参数名不是写死在每个页面中的固定列，而是由参数注册表统一描述道具归属、单位、精度和备注；重量模板保存参数值。

当前初始工作区包含：

- 26 个参数定义；
- 12 个重量模板；
- 47 个维度选项；
- 9 层顺序规则；
- 9 个词条；
- 4 个品质区间；
- 12 个系列/SKU 配方；
- 32 个候选组合；
- 1 张默认规则 DAG。

正式 SKU 和杆、轮、线明细由候选审核后产生，不在初始种子中预先固化。

### 2. 核心领域对象

- 参数定义：参数的稳定 key、显示名、道具类型、单位、精度和说明。
- 重量模板：钓法 × 大重量段的中性杆轮线基准，包含目标鱼重范围、标称鱼重和动态参数值。
- 维度选项：结构、材质、功能、性能、技术、系列六个维度中的一个可选项；每项拥有若干数值规则。
- 规则层：定义数值规则的确定执行顺序。
- 词条：同时拥有属性效果、被动语义和品质价值。
- 系列配方：规定允许的模板、结构、定位、技术、必带词条、可选词条池和候选数量。
- 候选装备：所有自动计算、比较、人工精调和筛选的中心草稿。
- 正式 SKU：候选发布时冻结的业务快照。
- 明细覆盖：在正式 SKU 的杆、轮、线明细上单独配置型号、名称和个别参数。
- 规则图与执行批次：保存跨步骤编排、运行状态和人工审阅快照。
- 工作区版本：保存完整配置状态及修改说明，并提供乐观冲突保护。

### 3. 数值规则采用“分层栈”

单件候选的确定性计算使用按 order 排序的规则层：

1. 基准模板；
2. 结构类型；
3. 类型材质；
4. 功能定位；
5. 性能定位；
6. 技术与特殊系列；
7. 系列共性；
8. 词条效果；
9. 手工精调。

规则支持六种操作：加、乘、覆盖、取最小值、取最大值、安全公式。公式由自定义解析器计算，不使用 eval。每次计算记录来源层、来源项、参数、计算前值、操作数和计算后值，形成可解释轨迹。

后层可以在前层结果上浮动，也可以通过 set 覆盖前层。手工精调永远位于最后，避免人工修正污染通用规则。

### 4. 跨步骤编排采用“有类型 DAG”

纯 Stack 无法表达条件分支、支路汇合、人工关卡和手动启动；任意 Graph 又可能产生循环和不确定执行。因此当前选择：

- 底层编排是禁止成环的有向无环图；
- 每条实际路径仍按拓扑顺序确定执行；
- 节点内部继续使用分层数值规则；
- 条件节点按候选行分别路由，而不是整批只能走同一分支；
- 汇合节点等待所需上游结束；
- 人工审阅节点生成可编辑中间表并暂停；
- 最终只把规则图或人工真正触碰的字段写回候选 overrides。

默认图为：

    装载中性模板
    → 一阶系数叠加
    → 词条效果与品质
    → 上下限约束
    → 特殊组合判定
       ├─ 特殊组合 → 强攻组合上/下调（手动）
       └─ 普通组合 ─────────────────┐
    → 支路汇合
    → 人工审阅中间表
    → 强度与覆盖验算
    → 下发候选池（手动）

默认特殊组合条件为：功能定位包含“障碍”、平台定位包含“巨物”或系列包含“巨”。特化规则增加杆轮拉力、杆耐力和杆重，同时轻微降低杆抛投，明确表达性能取舍。

图支持自动、手动、混合三种模式。节点状态包括 pending、ready、running、waiting_review、completed、failed、skipped。运行批次保留工作行、节点输入/输出行、条件命中情况和中间快照。

### 5. 词条与品质模型

词条分为两类：

- 直接属性词条：通过加、乘、覆盖等规则直接修改参数；
- 被动机制词条：表达抗冲击等不能完全压缩成单一数值的能力，也可以携带部分属性规则。

品质不是预先手填，而是由 SKU 已选择词条的总价值决定。当前算法为：

1. 过滤禁用词条并按基础分从高到低排序；
2. 同一能力轴的第 2、3 条以后按可配置系数递减计分；
3. 直接属性词条和被动词条分别乘以类别权重；
4. 已配置的协同词条对增加奖励；
5. 冲突词条对扣除惩罚；
6. 最终分不低于 0；
7. 按品质区间映射到品质 ID。

品质计算会输出每个词条的贡献、递减系数、协同奖励和冲突惩罚，便于解释。词条既影响属性又贡献品质分，SKU/系列决定拥有哪些词条。

### 6. 配方、候选和正式 SKU

系列配方是批量生成入口。系统对模板、结构、功能定位、性能定位进行受约束笛卡尔组合，叠加技术、必带词条和轮换选择的可选词条，并受 maxCandidates 限制。

候选生成后立即走完整计算，可在横向表格中比较、批量修改、入围、淘汰或发布。

发布正式 SKU 时：

- 组合 ID 沿用候选 comboId；
- 杆 ID 自动为“组合ID_R”；
- 轮 ID 自动为“组合ID_W”；
- 线 ID 自动为“组合ID_L”；
- 钓性、硬度、长度默认来自计算结果，也允许候选级覆盖；
- 杆、轮、线拉力和安全工作拉力被冻结到 SKU；
- 价格指数 = 品质区间价格指数 ×（1 + 特化维度数量 × 0.04）；
- 杆、轮、线明细可进一步定义型号、名称和局部参数覆盖。

### 7. 验算规则

当前安全工作拉力为：

    min(杆最大拉力 × 0.9, 轮最大拉力, 线最大拉力 × 0.35)

当前检查包括：

- 杆、轮、线最大拉力必须大于 0；
- 轮/杆拉力比建议在 0.55–1.20；
- 线/轮拉力比建议在 1.40–4.00；
- SKU 目标重量段必须覆盖模板标称鱼重；
- 饵重下限必须小于上限；
- 杆重、轮重必须大于 0。

错误、警告和信息级问题都进入候选/规则图验算结果，不直接静默丢弃。

### 8. 人工审阅与循环学习

规则图执行到 review 节点后冻结一张中间表。审阅者可以在杆、轮、线三个视图中修改参数，并填写备注；批准后才继续验算与输出。

人工改动以 patch/override 思路回写：只提交 touchedKeys，不把整张中间表覆盖回候选。

系统会统计多个候选相对模板的人工覆盖差值；同一参数出现至少两次精调时，生成“平均增量”规则建议。它目前只提出候选规则，不自动修改前层规则，符合“人审—总结—固化—再循环”的方向。

### 9. 工作台界面

当前主要模块包括：

- 装备生成总览；
- 重量模板；
- 分层规则；
- 规则图执行；
- 词条库；
- 品质评分；
- 系列/SKU 配方；
- 候选池；
- 正式 SKU；
- 杆、轮、线明细；
- 校验与规则学习；
- 版本记录。

编辑以密集列表、横向对比表和批量操作为主。规则图页面分“规则图编排”和“执行监控”：前者编辑节点、条件、规则和连线，后者展示当前节点、下一节点、就绪/运行/审阅/完成状态及中间审阅表。

### 10. 数据与部署

工作区以完整 WorkspaceState 保存，保存时携带 baseRevision；旧版本写入会返回冲突，要求刷新后合并。

目前保留两套后端适配：

- OpenAI Sites：D1 保存工作区与版本，R2 保存 Excel 原文件；
- Vercel：香港区域私有 Blob 保存工作区、历史快照和 Excel 原文件；使用强 ETag 条件写入防止并发覆盖，并使用支持强一致读取的 Blob SDK。

当前可访问正式地址：

- Vercel：https://tackle-forger-workbench.vercel.app
- Sites：https://tackle-forger-futou.futousteven.chatgpt.site

Vercel 已完成真实持久化验收：首页、状态 API、版本 API 均可用，保存后的 v2 能立即重新读取。

### 11. 当前必须保持的系统不变量

- 模板是中性基准，不应混入某个具体系列的特殊偏置。
- 所有数值变化必须可追踪到层、选项、词条、规则图节点或人工审阅。
- DAG 必须无环；支路执行和汇合结果必须确定。
- 审阅前的数据不能自动流向下游。
- 只有 touchedKeys 可以从规则图提交为候选覆盖。
- 正式 SKU 是发布快照；后续模板变化不应悄悄改写已发布 SKU。
- 版本冲突不能以“最后写入者覆盖一切”解决。
- 词条的属性效果与品质价值需要同时可解释。
- 规则学习只能先提出建议，不能未经人工批准自动污染通用规则。

## 合理推断

1. 当前实际上存在两个层次的执行器：单候选分层计算引擎，以及批次 DAG 编排引擎。二者共享规则操作，但仍有重复实现。长期应抽出唯一的 Rule Kernel，让普通重算、DAG 节点和规则学习都调用同一语义。
2. 规则图运行批次及其审阅快照已经接近正式的“设计实验记录”。它应成为未来复现、比较和回滚的主要审计单位，而不仅是临时 UI 状态。
3. 品质分目前主要回答“词条总价值”，不能完整回答“装备是否好用”。强度闭环、适配性、冲突和特殊机制可能需要独立于品质分的合格性判定。
4. 参数注册表是整个系统的真正基础设施。未来新增参数、改变单位或精度时，应优先通过参数元数据驱动公式校验、表格渲染和导入导出，而不是继续增加页面特例。
5. 当前公开 Vercel 站点没有真实的角色认证；只要知道网址就可能修改共享配置。用于正式团队生产前，需要补上身份、权限和审计主体。

## 未决问题

1. **唯一执行语义**：分层栈是否应被编译成 DAG，还是保持“单件计算栈 + 批次编排 DAG”两层架构？两层架构的共享 Rule Kernel 边界在哪里？
2. **规则版本化**：执行批次应该引用可变规则 ID，还是冻结规则内容和版本？历史批次如何保证以后仍能完全复现？
3. **汇合语义**：当两个分支修改同一参数时，应禁止、按优先级合并、要求显式 merge rule，还是产生人工冲突？
4. **审阅数据模型**：人工审阅应保存“完整快照”还是“基准快照 + patch”？批准、驳回、退回上游和二次审阅如何建模？
5. **品质定义**：品质是否只由词条价值决定？被动技能、属性词条、协同组合和负面词条是否应使用同一个总分，还是拆成预算、稀有度与有效性三个指标？
6. **品质目标约束**：系列配方中的品质目标是生成约束、筛选条件、发布门槛，还是仅用于提示偏差？
7. **SKU 身份与版本**：comboId 在配方、模板或规则变化后是否仍表示“同一个产品”？正式 SKU 是否需要 revision、世代或 variant identity？
8. **特殊规则治理**：第几层开始算“特化”？层级、优先级和覆盖权是统一数值，还是不同节点类型拥有固定权限？
9. **规则学习治理**：多少次人工精调、怎样的样本覆盖和置信度才允许建议升级为正式规则？谁审批，如何做回归比较？
10. **多人协作与安全**：正式使用时需要哪些角色（管理员、设计者、审阅者、只读者）？公开查看与配置写入是否应分离？
11. **被动技能执行**：抗冲击等非纯数值词条最终只用于描述和品质评分，还是需要进入战斗/钓鱼模拟器的事件规则系统？
12. **发布后的变更传播**：模板或规则修订后，已发布 SKU 是保持冻结、提示可升级，还是自动生成新修订候选？

## 下一轮最值得继续讨论的 3 个入口

1. **先确定唯一执行语义**：明确分层栈、Rule Kernel、DAG 节点和规则版本之间的正式关系；这是后续可复现性和维护成本的基础。
2. **再确定品质/词条的经济模型**：区分词条预算、品质、有效性与发布门槛，决定被动技能和协同冲突如何计价。
3. **最后确定人工审阅与规则治理**：设计 snapshot/patch、审批、回滚、规则学习晋级和多人权限，使“循环优化”真正可长期运行。
