## 1. 文档定位与权威边界

本文只描述交付Phase 4可能采用的并发目标架构、进入条件、迁移顺序和验证门槛，目的是为后续扩容留下一个可审查的设计基线。

本文不能单独授权以下任何变化：

- 取消或缩小当前工作区级单写锁；
- 新增运行时状态、数据库表、API、队列、锁服务或多节点部署；
- 改变Patch、UpgradeCandidate、RuleSetVersion或ConfigurationSnapshot的现行状态语义；
- 改变飞书写回、配置ID治理、Git ref CAS或浏览器本地文件恢复的现行门禁；
- 将尚未确认的开放问题固化为默认行为。

当前产品与领域行为始终由v3权威规范决定。Phase 4真正启动前，必须另立实现Issue，先更新v3和相应策略版本，再通过代码、迁移、故障注入和回退验证。若本文与当时的v3冲突，以当时的v3为准。

本文中的“交付Phase 4”是第25.1节产品交付分期，不是第17节“当前实现迁移”中的“阶段4”，二者不得混用。

## 2. 为什么放在Phase 4

当前工作区级单写锁刻意优先保证：

- 关键共享状态只有一个在途写者；
- fencing token、幂等回读和失败恢复的责任边界清楚；
- 飞书、配置治理、Git ref和浏览器本地文件等不同副作用不会被表面并行掩盖；
- 现阶段可以先证明确定性、可追踪、可重放和历史冻结。

资源级并发只有在真实使用数据证明工作区锁成为瓶颈后才有收益。在此之前拆锁会增加死锁、饥饿、跨资源部分成功、未知外部结果和多节点协调成本，却不能改善领域正确性。因此一期、1.5期、二期和三期均不因本文改变现状；Phase 4也不能仅因“技术上可以”而自动启用。

## 3. Phase 4入口门槛

只有以下条件全部满足，才允许创建“启用资源级并发”的实现提案：

1. 一期至三期的适用能力已按各自验收条件稳定运行，当前工作区单写锁及恢复路径已有生产证据。
2. 监控数据能够证明存在持续的锁等待、队头阻塞、吞吐或延迟问题，并能区分真正共享资源与无关操作。
3. 每类关键操作已有统一的命令记录、幂等结果、恢复状态和可回读证据；未知结果不会被标成成功、失败或取消。
4. 目标数据库和协调基础设施能提供所需的事务、条件更新、唯一约束、单调token和租约连续性。
5. 资源身份、资源集合展开、稳定排序和提交时重验已经形成版本化契约，不使用展示文案或字符串拼接猜身份。
6. 随机交错、租约过期、响应丢失、重复请求、死锁顺序、部分外部写入和历史冻结测试全部通过。
7. 存在工作区级回退开关，能够在不迁移或改写历史事实的前提下恢复为单写模式。
8. 独立策略版本、实现Issue、迁移方案、观测窗口和明确授权已经完成。

任一门槛缺失时保持当前工作区级单写锁。性能预估、压测中的理想吞吐或开发环境中的偶然并发不能替代生产形态证据。

## 4. 目标模型：事实、命令、协调、副作用四层分离

Phase 4目标不是“去掉锁”，而是让每类一致性责任只有一个权威层。

### 4.1 不可变领域事实

Model、Patch、RuleSetVersion、ConfigurationSnapshot、顶点集合和发布证据继续使用不可变revision或不可变快照。可变头指针只负责指出当前revision，不能改写历史Payload或hash。

领域不变量：

- 一个稳定实体在任一提交点最多有一个合法当前head；
- 已发布ConfigurationSnapshot内容、引用和hash永久不变；
- 永久ID、名称与reservation ledger约束不能因重试或迁移被复用；
- 同一输入、策略版本和有序操作产生相同结果与Trace。

### 4.2 命令并发控制

每个写命令都应携带完整、可验证的期望基线，并以条件更新推进目标head。规划中的统一命令信封至少包含：

```ts
interface CommandEnvelopeV1 {
  commandId: string;
  commandType: string;
  workspaceId: string;
  subjectRef: EntityRef;
  expectedRevisionId: string;
  policyVersionRefs: string[];
  inputHash: string;
  payloadHash: string;
  idempotencyKey: string;
  requestedBy: string;
}
```

该结构只是Phase 4规划，不是现行API。正式定义前仍须确定字段闭集、规范序列化、哈希算法、幂等作用域和兼容迁移。

命令提交必须满足：

- 期望revision、策略版本、资源token和输入hash在实际提交点仍有效；
- 相同幂等键和相同完整Payload返回第一次正式结果；
- 相同幂等键配不同Payload必须拒绝；
- 冲突方不留下半revision、半操作组、半证据或提前推进的head。

### 4.3 资源协调

只有确实共享同一提交决策的操作才竞争同一资源。资源租约提供顺序与fencing，不取代数据库事务、唯一约束或expected revision。

资源身份必须是版本化的结构化值。候选资源包括：

```text
workspace/{workspaceId}/rules
workspace/{workspaceId}/patch/{patchId}
workspace/{workspaceId}/snapshot-publication
five-axis/{vertexGroupKey}
config-id/{rangeId}
config-target/{repositoryId}/{authoritativeRef}
local-export/{bindingId}/{environmentId}/{channelKey}
```

这些路径仅用于说明资源类别，不能直接作为字符串拼接后的正式锁键。正式协议应使用带`resourceType`和类型化字段的闭集Schema，并采用与v3一致的逐分量稳定排序。

一次操作必须在准备阶段展开完整资源集合，去重后按全局稳定顺序一次取得全部资源；禁止运行中锁升级。每份租约分配严格单调且永不复用的fencing token，所有受保护提交点都重新验证token。无法证明token连续性时fail-closed。

### 4.4 外部副作用与恢复

外部副作用必须由持久化Operation承接，分开表达准备、执行、回读、激活与恢复。外部系统返回超时或连接中断时，结果是“未知”，不是自动失败或取消。

核心原则：

```text
准备并冻结输入
→ 取得资源租约
→ 写入持久化Operation / outbox
→ 执行外部写入
→ 以幂等身份回读
→ 保存验证证据
→ 在当前token下激活领域结果
```

旧token不得写入成功证据。新token也不能越过同一目标尚未消歧的旧操作。

## 5. 领域状态与操作状态正交

Patch业务生命周期继续只使用v3第14.2和24.11节的大写`PatchState`。基线变化使当前Patch revision进入`REBASE_REQUIRED`；成功Rebase创建新revision，状态最多为`PENDING_REVIEW`。

Phase 4若需要持久化Rebase执行进度，应使用独立的操作实体，而不是把`REBASING`加入PatchState：

```ts
interface RebaseOperationV1 {
  operationId: string;
  patchId: string;
  sourcePatchRevision: string;
  sourceBaseRefs: string[];
  targetBaseRefs: string[];
  expectedHeadPatchRevision: string;
  inputHash: string;
  idempotencyKey: string;
  state:
    | "PREPARED"
    | "RUNNING"
    | "VERIFYING"
    | "COMMITTED"
    | "FAILED"
    | "SUPERSEDED"
    | "RECOVERY_REQUIRED";
}
```

该枚举同样只是规划候选。正式状态机必须在Phase 4实施Issue中定义转换、终态、重试、清理和迁移。

关键边界：

- 运行中的Operation不改变原Patch revision内容或hash；
- 目标基线或Patch head并发变化时，Operation进入`SUPERSEDED`或失败，不能覆盖新head；
- 只有同一事务成功写入完整新Patch revision、操作组、幂等记录与审计后，Operation才能记录`COMMITTED`；
- 未知提交结果进入`RECOVERY_REQUIRED`并按幂等键回读；
- Snapshot、历史Patch引用和PatchSetHash不因Operation状态变化而改变。

同样的正交原则适用于发布、飞书写回、配置导出和恢复：领域产物的状态不能兼任长任务进度条。

## 6. 何时可以拆分工作区锁

候选操作只有在以下共享关系全部不存在时，才可能独立提交：

- 相同实体head或同一不可变产物的创建决策；
- 相同永久ID区间、名称唯一约束或reservation ledger游标；
- 相同五维`vertexGroupKey`或通过迁移边相连的组；
- 相同物理Git ref，即`repositoryId + authoritativeRef`；
- 相同本地导出逻辑目标；
- 相同飞书目标或尚未消歧的外部幂等操作；
- 其他必须在同一事务或同一激活点决定的共享资源。

若资源分析无法证明独立，默认归入工作区资源并继续串行。不能用不同页面、不同显示名称、不同`targetEntryId`或不同操作者证明独立。

Phase 4早期建议先做“影子资源分析”：仍持有工作区锁执行，只记录如果按资源拆分会取得哪些锁、是否出现遗漏或排序冲突。影子结果稳定后才允许小范围启用真实资源锁。

## 7. 用依赖图确定事务边界

v3第21.3和25.3节已经规定：五维候选变更按受影响`vertexGroupKey`建立依赖图，每个连通分量分别形成原子事务。Phase 4可以把这一模式推广到确实存在跨资源迁移的批量操作：

1. 把全部受影响资源作为图节点；
2. 同一变更必须同时修改两个或多个资源时，在节点之间建立无向边；
3. 每个连通分量是最小原子提交边界；
4. 分量内按结构化资源键稳定排序并一次取齐租约；
5. 分量内全部成功或全部回滚；
6. 没有依赖路径的分量可以分别成功、失败和重试。

可能适用的后续场景包括Snapshot批次、Patch批量复核、RuleSet重算和多目标导出。是否真的适用必须逐场景证明；不能因为“五维已经使用依赖图”就把图算法自动套到所有业务。

## 8. 三类外部写入的不同协议

### 8.1 服务端可达目标

飞书等服务端可达、但不能原生验证Tackle Forger fencing token的系统继续使用持久化fenced outbox：

- 数据库事务冻结命令、目标、幂等键和token；
- worker在调用前重验token；
- 外部写入后按幂等身份回读；
- 回读证据在当前token下持久化；
- 证据完整后才激活后续领域状态。

低token操作结果未知时，高token操作不得越过同一目标。

### 8.2 Git ref

Git写入的物理锁身份固定为`repositoryId + authoritativeRef`，不是逻辑环境、渠道或目录条目。推进ref必须同时满足：

- 当前治理租约和fencing token有效；
- expected old OID与实际ref一致；
- 所有逻辑别名对同一物理ref声明相同OID；
- 写入网关或受保护ref协议能够拒绝绕过路径；
- 写后解析新OID并保存回读证据。

任一条件不能强制执行时保持`CONFIG_TARGET_SERIALIZATION_UNAVAILABLE`，不得用资源级并发降低现行门禁。

### 8.3 浏览器本地文件

浏览器本地文件系统不能真正执行服务端fencing。即使Phase 4拆分了服务端资源锁，本地写入仍必须依赖：

- 每文件基线hash与mtime；
- 备份和恢复Manifest；
- 每次写入前重新验证基线；
- 逐文件写入和逐文件回读；
- 部分成功后的回滚或前向协调；
- 未知终态进入`RECOVERY_REQUIRED`；
- 恢复完成前阻止相同逻辑目标的新正式写入。

服务端token只限制成功证据，不能声称撤销已经写入本机的字节。

## 9. 公平性、背压与取消

资源级并发不能只追求吞吐，还必须避免长期饥饿和大操作占满所有资源。

Phase 4策略至少需要版本化定义：

- 每个资源的FIFO或等价公平队列；
- 已进入提交阶段的操作不得被后来的短操作绕过；
- 单操作最大连通分量规模、最大持锁时间和最大外部调用数；
- 工作区、用户和操作类型的在途上限；
- 超限时是拒绝、排队还是要求用户拆批；
- 租约心跳、过期和恢复优先级；
- 取消允许发生的最晚阶段。

取消只在尚未产生外部副作用或能够证明副作用未发生时结束为`CANCELLED`。外部结果未知、部分本地写入或数据库提交结果未知必须进入恢复状态，不能为了界面简洁标成已取消。

## 10. 观测与上线判定

资源拆分前后至少观测：

- 工作区锁等待时间、持有时间和队列长度；
- 每类结构化资源的竞争率；
- 因expected revision、token或基线变化导致的冲突率；
- 幂等重放和未知结果回读次数；
- `RECOVERY_REQUIRED`数量、恢复时长和人工介入率；
- 每个依赖图连通分量的节点数和事务时长；
- 公平队列等待分布与饥饿告警；
- 回退到工作区锁的次数与原因。

启用顺序应从低风险、天然独立、无外部副作用的资源开始。每一步都要有观察窗口和自动/人工回退条件，不能一次性把所有关键操作从工作区锁切换出去。

## 11. Phase 4内部迁移顺序

### 4A：统一记录，仍保持工作区锁

- 统一命令信封、Operation、幂等结果和恢复证据；
- 为现有操作记录结构化资源集合；
- 不改变并发度或外部行为。

### 4B：影子分析与度量

- 在工作区锁保护下运行资源展开、排序和依赖图；
- 对照真实读写集合检查遗漏；
- 收集竞争、延迟和恢复数据；
- 任何不确定资源仍归入workspace。

### 4C：单节点资源级并发

- 通过策略版本和工作区功能开关，只启用已证明独立的资源类别；
- 保留数据库唯一约束、expected revision、fencing和外部恢复；
- 支持立即回退到工作区锁；
- 全程记录影子结果与实际提交结果差异。

### 4D：多节点协调

- 在4C长期稳定后，才考虑多节点队列、工作区分片或资源协调服务；
- 跨工作区的全局资源仍由全局协调器保护；
- 节点失联、时钟漂移、网络分区和协调服务恢复都必须有故障注入证据；
- 多节点不是Phase 4启动时的默认交付。

每一子阶段都需要独立Issue、迁移、验证和回退决定。完成4A不自动授权4B，完成4B也不自动授权真实拆锁。

## 12. 最低验证矩阵

正式实现至少覆盖以下场景：

| 类别 | 必测场景 |
| --- | --- |
| 正常 | 两个无共享资源的命令可以并行提交，结果与串行执行一致 |
| 边界 | 空资源集、单资源、最大分量、最大token、租约临界过期均fail-safe |
| 冲突 | 同一head、永久ID、顶点组、Git ref或本地目标最多一个冲突提交成功 |
| 死锁 | 随机打乱请求顺序后仍按结构化全序取锁，不发生锁升级或循环等待 |
| 幂等 | 提交后响应丢失、重复请求、同key不同Payload都得到唯一正式结果 |
| 租约 | 旧token在租约过期和新token发放后不能写业务状态或成功证据 |
| 依赖图 | W1↔W2交换、跨三组链式迁移同分量原子提交；无依赖W5独立 |
| Git | 不同逻辑别名指向同一物理ref时竞争同一锁；OID冲突fail-closed |
| 本地文件 | 第二个文件写入失败、页面断线、部分回滚失败均进入可恢复终态 |
| 外部系统 | 调用已生效但响应丢失时先回读，不重复产生副作用 |
| 公平性 | 长操作与连续短操作并存时没有永久饥饿，背压按策略执行 |
| 历史冻结 | 任意冲突、恢复、回退和迁移都不改变历史Snapshot、Patch与hash |
| 二次迁移 | 数据迁移第二次执行为no-op，未知字段和无法表达数据进入人工复核 |
| 历史检查 | 随机操作历史通过线性化/不变量检查器，不出现两个合法head或部分分量 |

除确定性用例外，还应使用随机交错、属性测试和故障注入覆盖最小复现场景，并保存seed、操作序列、策略版本和最终状态hash。

## 13. Phase 4启动前仍需确认的问题

以下问题保持开放，本文不提供隐藏默认值：

- 首批允许拆分的资源类别和业务操作；
- 生产数据库、租约协调器和队列实现；
- 公平队列、最大分量、持锁时长和背压阈值；
- Operation状态机的完整转换、保留与清理策略；
- 跨工作区全局资源清单；
- 多节点分片键和再均衡协议；
- 回退开关的触发阈值、观察窗口和授权人；
- 是否以及何时从4C进入4D。

这些决定必须在Phase 4实施Issue中以生产形态数据、故障模型和可回退方案逐项确认。缺失时继续使用工作区级单写锁。
