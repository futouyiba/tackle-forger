# 从“交叉相乘”到可治理的钓具生产系统

> 文档定位：面向最初提出“基础模板 × 多层系数”方案的设计说明与沟通材料  
> 日期：2026-07-22  
> 注意：本文解释设计演进，不替代 [`tackle-forger-development-spec-v3.md`](./tackle-forger-development-spec-v3.md)。文中“已经完成”主要指需求澄清和方案定型；具体功能是否已经上线，以当前开发与验收状态为准。

## 一、原始思路没有被推翻，而是成为了系统的计算内核

最初的想法非常清楚：先定义不同重量段的基础钓具，再让钓法、类型和功能定位通过系数参与计算。

```text
基础重量段模板
× 钓法系数
× 类型系数
× 功能定位系数
= 某一类钓具的属性标杆
```

这个方向解决了最关键的起点问题：不再逐件手填所有钓具，而是通过少量基础模板和可复用规则，系统化地生成大量合理结果。它让钓具设计第一次具备了规模化、统一性和调整全局平衡的可能。

后续的设计工作并不是改掉这个想法，而是继续追问：规则越来越多时怎样避免组合爆炸？某个结果不合适时改哪里才不会误伤其他装备？人工调整后，上游规则变化还能不能重新计算？怎样追踪最终数值的来源？怎样表达同一系列的多个重量和每个重量下的多个型号？怎样保证已经发布到游戏中的配置不会被新规则悄悄改变？

因此，我们把最初的“计算公式”扩展成了一套完整的钓具设计、校验、审阅和发布系统。

## 二、最大的升级：从一个公式变成一条可治理的生产链

最终形成的主链路是：

```text
飞书规则源
→ 已发布规则版本
→ 结构标杆派生
→ Series 系列
→ 离散重量 SKU 抽屉
→ 可购买 Model
→ 配置校验
→ ConfigurationSnapshot 发布快照
→ tackle / item / store 配置表
```

这里最重要的变化，是把四种性质完全不同的数据分开：

1. **源规则**：策划在飞书维护的基础模板、系数、定位和词条定义。
2. **派生结果**：系统按固定顺序计算出来、可以重建的只读结果。
3. **人工 Patch**：针对某个中间层、Series、SKU或Model的显式修正。
4. **发布快照**：经过校验后交给游戏使用、发布后不可被静默改写的正式结果。

这项拆分解决了早期方案最危险的混淆：如果把计算结果直接当源数据修改，我们很快就无法判断一个数值究竟是规则算出来的，还是某个人手工改过的；更无法在规则升级后安全地重新生成。

## 三、对模板层级的优化：只展开有限、稳定的结构维度

### 3.1 钓法和类型分成两个规则层

我们确认钓法与类型不能合并成一个不可拆分的大表。

- 钓法描述玩法环境和基础倾向，例如路亚、浮钓。
- 类型描述结构差异，例如直柄、枪柄、纺车轮、水滴轮。

两者可以在同一个页面步骤里操作，但底层仍然分开计算。修改“路亚”的通用规则时不需要复制所有类型；修改“枪柄竿”的结构规则时，也不会把路亚规则重新写一遍。

这是典型的“界面可以合并，领域语义不能混合”。它减少重复数据，也保留了未来扩展新钓法、新类型的能力。

### 3.2 功能定位显式描述优势与代价

远投、精细、大物搏鱼等功能定位，不再只是一个标签，而是一组明确的属性操作。例如大物搏鱼可能包含：

- 拉力增加20%；
- 自重增加20%；
- 长度增加10%；
- 抛投精准度降低30%；
- 抛投距离降低10%。

这样，功能定位真正参与了钓具形态的演绎。同时系统知道“自重增加”虽然数字变大，却是一项代价，而不是收益。

### 3.3 性能、品质、材料和词条不进入结构标杆的笛卡尔积

这是后续澄清中非常重要的一次收敛。

如果把性能定位、品质、材料和各种词条也全部放入预生成模板，组合数量会迅速接近无限：每增加一个词条或等级，模板数量都成倍增长。更严重的是，结构身份和商品强化会混成一件事。

因此，结构标杆只由有限维度决定：

```text
部位 + 重量段 + 钓法 + 类型 + 功能定位
```

品质在编辑Series前确定；性能、材料和词条在具体产品生成阶段应用。即使某个词条也提供“拉力+20%”，它改变的是最终Model属性、价值分和兼容结果，不会反过来要求系统重新选择结构标杆。

这项设计把“它本质上是什么钓具”和“这件具体商品获得了哪些强化”彻底分开，既控制了规模，也避免了循环计算。

## 四、标杆选择的优化：先完成结构演绎，再按拉力找最近结果

最初容易被误解成“只按原始重量段找模板”。后来我们明确：重量段本质上是基础拉力段，必须先把钓法、类型和功能定位的影响计算完，再在同类结构标杆中匹配。

例如，一个基础标杆的拉力范围是0.8–3.0kg，大物搏鱼功能使拉力增加20%，那么派生后的对应范围会变成0.96–3.6kg。新的Series重量规格，应当与这批已经完成结构演绎的标杆比较，而不是回到0.8–3.0kg的原始模板比较。

搜索范围必须先严格限定为：

```text
相同部位
+ 相同钓法
+ 相同类型
+ 相同功能定位
```

然后使用比例距离寻找拉力最接近者：

```text
distance = abs(ln(targetPullKg / derivedPullKg))
```

使用比例距离而不是线性距离，是因为0.1kg到100kg横跨多个数量级。比例距离能让轻型和重型装备获得一致的“相对接近”语义。

我们同时确认：不做连续插值；1.5kg和1.8kg可以命中同一个结构标杆，但仍然是同一Series下两个独立SKU；Affinity不参与结构标杆选择；词条和后置Patch也不反向改变标杆。

这样得到的结果稳定、容易解释，也符合策划“从已有标杆中找最接近装备”的工作习惯。

## 五、每一层都能调整，但调整不再破坏模板

### 5.1 为什么要增加中间层 Patch

在实际检查派生结果时，问题不一定只发生在最终Model。例如“钓法 × 重量段”的结果整体有偏差，“重量段 × 钓法 × 类型”的枪柄竿普遍过重，某个功能定位在多个重量段上都缺少代价，或者只有某一个Series、某一个重量SKU、某一个Model需要例外。

如果只能改最终值，同样的问题会在几十个产品上被重复修补；如果直接改派生模板，又会失去来源和重算能力。因此我们设计了两类Patch：

- **DerivationLayerPatch**：修正共享的中间派生层，例如钓法层、类型层或功能层结果。
- **Series/SKU/Model Patch**：处理具体系列、重量抽屉或型号的个案差异。

### 5.2 Patch不是一份被覆盖后的最终值

每条Patch会记录：

```text
作用对象
+ 参数
+ before
+ operation
+ operand
+ after
+ 修改理由
+ 作者与时间
+ 基线规则版本
+ 完整来源 Trace
```

操作也不是只有“强行设值”，而是区分：

- `multiply`：按比例修正，适合跟随新基底重放；
- `add`：固定补偿，适合量纲明确的差值；
- `set`：锁定绝对值，上游变化后必须人工复核；
- `min/max`：声明边界。

这让人工经验第一次成为可保存、可解释、可重新应用的数据，而不是散落在表格里的神秘数字。

### 5.3 Patch解决的三个核心问题

第一，**控制影响范围**。改共享规则、改整个Series、改一个SKU、改一个Model，是四种完全不同的行为，系统会让用户显式选择。

第二，**支持安全升级**。当上游规则变化时，系统可以在新基底上重放`multiply`和`add`，把`set`送回复核，而不是把所有人工调整一起丢掉。

第三，**把个案经验变成规则候选**。多个共享中间层出现相同修改时，可以形成飞书规则变更草稿；单个Series、SKU或Model的例外则保留为本地Patch，不污染通用规则。

通用Patch可以由策划人工确认后回写飞书，但“回写”“拉取”和“发布新规则版本”仍是三个显式步骤，防止循环同步和误发布。

## 六、从“生成一件装备”升级为Series、SKU抽屉和Model三层商品结构

早期方案主要关注属性怎么生成，后续我们补齐了游戏中真正需要的产品结构。

### 6.1 Series承载共同概念

Series表示一组概念一致的钓具。它拥有稳定身份、统一品质、钓法、类型、功能定位、核心词条和设计方向。

一个Series可以规划多个离散重量，例如1.5kg、3.5kg、8.2kg。同一Series内目标拉力必须唯一，因此这些重量分别生成三个SKU抽屉，而不是被合并成一个连续区间。

### 6.2 SKU是抽屉，不是玩家购买对象

SKU代表Series在一个目标重量上的产品抽屉，例如“青芦·远投 / 1.8kg”。它承载这一重量规格的标杆命中、Patch、候选和校验上下文。

### 6.3 Model才是最终型号

一个SKU抽屉中可以有多个实际Model，例如快调性短竿、慢调性长竿、不同轮体尺寸或线材组合。玩家最后选择和购买的是Model。

这个澄清避免了把“重量规格”和“可购买商品”混为一谈，也让同一重量下的不同手感方案拥有正式位置。

### 6.4 名称用于人看，稳定ID用于系统关联

名称可以改、可以重复、也可能因为文案调整发生变化，因此不能承担对象关联。Series、SKU、Model都使用稳定ID和revision追踪。

再生成时，系统通过父级稳定ID、目标拉力和对象身份键找到新旧对应关系，而不是猜名字。这样既允许改名，也保证重复生成不会创建一批无法对齐的“新对象”。

## 七、自动化与人工介入之间做了更成熟的平衡

最初容易走向两个极端：每一步都强制人工确认，效率太低；或者全部自动生成，设计师来不及发现问题。

最终规则是：

- 默认自动批量生成Series下的多个SKU及Model候选；
- 用户可以主动限制批量生成范围；
- 普通阶段默认不阻断；
- 用户显式介入，或阶段策略为`REVIEW_ON_CHANGE`时才暂停；
- 人工修改后，受影响的下游结果标记为`DIRTY`或“有升级候选”；
- 同输入、同规则版本、同显式seed必须得到相同结果；没有随机需求时不强行使用seed。

工具平时像一条自动生产线，需要时又能在任意关键层停下来检查。修改上游后，系统不会假装下游仍然有效，而是明确指出哪些结果需要重新生成或复核。

## 八、把“合理”拆成四套互不冒充的判断

讨论中另一个重要成果，是没有用一个模糊总分决定所有事情。我们把不同问题拆成四套独立语义：

1. **硬兼容**：物理上、结构上能不能组合、生成或发布。
2. **Affinity Score**：虽然能用，但玩法和设计方向是否契合，用于软排序和解释。
3. **系列不变量**：同一Series内部是否仍然是同一个产品概念。
4. **发布检查**：版本、Trace、跨表关联和正式配置是否完整。

Affinity再高也不能覆盖硬不兼容；AI建议也不能更改这些结论。这样设计避免了“总分很高，所以物理不兼容也通过”之类的错误。

所有问题统一成`ValidationIssue`，并携带严重程度、证据和可执行的`ActionLink`。用户不仅看到“错了”，还能知道为什么、影响谁、可以去哪里修复。

## 九、补上系列不变量，避免一个系列逐渐失去自己的身份

系列不仅是同一个名字下的一组产品，还必须保持概念一致。系统会检查：

- 钓法、类型、功能定位和品质是否一致；
- 同一Series中的目标拉力是否唯一；
- 核心词条是否覆盖全部SKU和Model；
- 禁用词条是否被错误加入；
- 更高重量的承载能力是否反向下降；
- Model的长度、调性变化是否仍在Series允许范围内；
- 优势与代价方向是否符合系列叙事。

例如“大物搏鱼”系列可以更重、更强、抛投更弱；不同重量的幅度可以变化，但不能有一个型号突然变成“超轻远投”而仍然假装属于同一Series。真的越过系列包络时，正确动作是创建新Series，而不是继续叠Patch掩盖概念变化。

## 十、品质、词条、价值分和定价的顺序被重新理清

我们最终明确了商品设计顺序：

```text
先确定品质
→ 编辑Series
→ 生成SKU与Model
→ 选择词条/技术
→ 计算价值分
→ 校验品质区间
→ 形成定价参考
```

品质固定映射为C/绿、B/蓝、A/紫、S/橙。`functionIntensity`只是功能专精强度，不是品质。技术是词条组合包，不能一边由技术增加属性，一边让技术内部词条再次增加同名属性。

选完词条后，系统计算并留存价值分：它用来检查这件装备是否符合当前品质的价值区间，也为价格计算提供依据，但不会反向偷偷改变品质。

被动词条在当前工具中只保存、展示和计分，不执行钓鱼模拟器逻辑。这是一次重要的边界控制：工具负责生产配置，不冒充战斗模拟器。

## 十一、所有最终数值都有统一Trace，不再依赖“谁还记得这个公式”

我们为属性来源定义了统一轨迹。任意最终属性都可以逐层展开：

```text
重量基础模板
→ 钓法贡献
→ 类型贡献
→ 功能定位贡献
→ 性能/词条贡献
→ Series Patch
→ SKU Patch
→ Model Patch
→ 最终值
```

每一段Trace都包含来源对象、规则版本、before、operation、operand、after、警告和可执行动作。

它带来三种价值：策划能看懂装备为什么变成这样；发现异常时能定位到真正出错的规则层；几个月后仍能复现当时发布的结果，而不依赖人员记忆。

## 十二、飞书从“共享表格”升级为唯一规则源，但同步保持显式可控

飞书电子表格继续是策划最方便的规则编辑入口，但工具不再把“表格里最新看到的内容”直接当正式规则。

完整链路被明确为：

```text
工具内规则草稿
→ 人工确认回写飞书
→ 技术回读确认
→ 提示远端有新变化
→ 用户显式拉取
→ 形成 RuleSet 草稿
→ 校验
→ 人工发布 RuleSetVersion
```

回写不等于拉取，拉取不等于发布。飞书变化也不会自动进入工具。

同时，工作表、规则实体、Series原型都拥有机器稳定ID。名称可以被策划调整，但系统不会因为改名就把旧实体误判成新实体。新增且缺少机器ID的飞书行会进入待确认状态，而不是按名称自动猜测关联。

## 十三、发布不再只是“导出一个Excel”

最后两步被补全为正式的配置交付流程：

1. 为合格的Model批量冻结`ConfigurationSnapshot`；
2. 向设计人员本地配置Git仓库的目标环境和渠道写入`tackle`、`item`、`store`三张表。

`store`必须生成，并包含上架开关。dev、test等环境拥有各自的`configs`目录和`config.toml`；每个环境下又可以选择1001或其他channel。

写入前必须预览差异，检查三张表之间的稳定ID和配置键关联，根据目标环境的TOML验证跨表关系，对无法关联的内容明确阻断，并在确认目标后备份和原子替换。

草稿不需要逐个手工冻结。发布时可以批量创建快照：未变化的复用旧快照，合格的新revision创建新快照，被阻断的对象跳过并汇总报告。

最重要的是，已发布快照永远不被上游变化静默重算。新规则只会产生`DIRTY`、Patch rebase或`UpgradeCandidate`，让用户比较旧值、新值和原因后决定是否发布新版本。

## 十四、界面设计也从“表单工具”升级为数据驾驶舱

为了让复杂系统仍然可用，界面采用高密度数据驾驶舱，而不是把用户困在一连串孤立表单中。

- **钓具系列甘特图**：按重量轴展示Series规划和离散SKU分布；它是界面心智模型，不是新的领域实体，也不表示连续插值。
- **SKU抽屉与Model右侧预览层**：从Series逐步深入实际型号，不丢失当前上下文。
- **属性来源表**：横向比较基础模板、各规则层和各级Patch。
- **五维属性图**：既能把竿、轮、线放在同图观察匹配关系，也能比较多个同部位钓具。
- **生成Model候选**：原“候选池”被收敛为Series/SKU上下文中的一次动作，避免它被误认为正式领域对象。
- **AI评估与建议**：规划为二期辅助能力，只能引用证据、预览建议和生成Patch或飞书规则提案草稿，不能自动应用、发布或推翻硬规则。

这套界面设计的目标不是“看起来复杂而专业”，而是让设计师在同一视野中回答：它来自哪里、改了什么、擅长什么、付出了什么代价、是否兼容、能否发布。

## 十五、需求澄清过程中最有价值的决定

| 决定 | 解决的问题 | 长期价值 |
| --- | --- | --- |
| 钓法与类型分层 | 避免规则重复和耦合 | 可独立扩展新玩法与新结构 |
| 结构标杆不包含性能、品质、材料和词条 | 避免组合爆炸 | 保持有限、稳定、可缓存的模板空间 |
| 先演绎结构，再按比例拉力距离匹配 | 避免选错标杆 | 符合真实钓具定位且跨数量级稳定 |
| 不连续插值 | 避免生成不存在的“混合标杆” | 结果可预测、可解释、可人工固定 |
| 各层使用Patch，不覆盖派生结果 | 避免人工调整失踪或污染源规则 | 支持重放、审计和安全升级 |
| 通用Patch与个案Patch分治 | 避免一个特例改坏全局 | 经验可沉淀，例外不污染规则库 |
| Series含多个离散SKU，Model才可购买 | 避免商品层级混乱 | 支持同重量多型号和稳定配置关联 |
| 硬兼容、Affinity、不变量、发布检查分开 | 避免模糊总分越权 | 每种结论都有清晰职责和阻断边界 |
| 优势和代价都进入计算 | 避免“所有属性只涨不降” | 支持真正的横向取舍与平衡 |
| 先定品质、后选词条并留存价值分 | 避免品质和词条互相倒推 | 定价、稀有度和设计审核有共同依据 |
| 显式拉取、版本化发布 | 避免飞书修改直接污染生产数据 | 可回滚、可复现、可审计 |
| 冻结发布快照 | 避免线上商品被新规则悄悄改变 | 保护历史版本和正式配置安全 |

## 十六、这套设计真正优秀的地方

如果只看功能清单，系统增加了模板预览、Patch、SKU抽屉、Model、兼容校验、五维图、价值分、定价、飞书同步、配置导出和发布快照。但更重要的提升其实来自设计思路。

### 16.1 把“生成”和“治理”放在同一个系统里

很多工具只能生成数据，却不能解释、修正和发布；另一些工具只能审核最终表格，却不知道数据怎样产生。Tackle Forger把生成、人工判断、版本和发布连成了一条链。

### 16.2 把自动化设计成可接管，而不是不可控

默认批量推进保证效率；阶段检查点、DIRTY传播和Patch让人能在必要时接管。自动化不是替代设计师，而是让设计师把精力放在真正需要判断的地方。

### 16.3 把错误修在正确的层级

共享偏差修中间层，系列特点修Series，重量差异修SKU，型号差异修Model。这个原则让系统不会随着时间推移变成一堆无法解释的最终值覆盖。

### 16.4 把每次发布当作可复现的产品版本

规则会变化、策划会改想法、Patch会重新基准化，但已经发布的配置保持冻结。这使工具不仅能“算出今天的结果”，还能够解释半年前为什么发布了那个结果。

### 16.5 把尚未决定的事情保留为配置，而不是过早写死

性能强度命名、Patch阈值、五维图轴、AI审核权限、定价曲线等仍在开放的事项，都没有被开发自行固化。系统为未来决定留出接口，同时不让未确认假设偷偷变成产品规则。

## 十七、可以怎样向最初提出方案的人概括

> 最初“基础重量模板乘以钓法、类型和功能系数”的想法，仍然是今天系统最核心的计算方式。我们后来做的工作，是给这个内核补上了可长期使用所需要的结构：只让有限的结构维度进入标杆，避免词条组合爆炸；先完成结构演绎，再按比例拉力距离选最近标杆；允许在中间层、Series、SKU和Model分别打Patch，又不破坏源模板；把硬兼容、软适配、系列一致性和发布检查分开；最后用稳定ID、完整Trace、显式飞书同步和不可变快照，把结果安全地写进游戏配置。于是它不再只是一个交叉相乘的小工具，而成为了一条可解释、可人工接管、可持续演进的钓具配置生产线。

## 十八、结语

这次设计演进的价值，不是单纯“多加了很多功能”，而是把原始创意保护了下来，并补齐了它走向真实生产所需要的边界。

交叉相乘负责提供规模化和一致性；分层规则负责控制复杂度；最近标杆负责稳定映射；Patch负责承载人工判断；校验负责阻止不合理组合；Trace负责解释；版本和快照负责保护正式结果；飞书与配置导出则把策划工作真正连接到生产环境。

最终形成的不是一个替人填表的计算器，而是一套能够沉淀设计知识、支持团队协作、经得起规则变化并保护已发布内容的钓具生产系统。
