## 1. 背景与产品判断

Tackle Forger 已经具备大量严谨的领域规则、版本、Trace、Patch 和发布治理，但这些能力主要沉在表格、状态和详情中。用户能看到结果，却不容易在短时间内感受到：

- 数据来自哪里；
- 系统按什么顺序处理；
- 哪些来源改变了结果；
- 为什么数字最终变成现在这样；
- 当前结果是草稿、可复核结果，还是已冻结事实。

动态表现的首要任务不是装饰界面，而是把这条因果链浮到水面。它应当形成“严谨工作台里的高速结算感”：有速度、有完成感，但不采用赌场式刺激、手游式奖励皮肤或与现有数据工作台割裂的视觉语言。

独立审计确认当前方向成立：约 2.2 秒的属性结算不再寡淡，且与工作台融合良好。主要问题不是速度过快，而是同一拍中飞卡、增量、主数字、链路计数、证据和右侧解释同时争夺注意力，最终完成态也缺少独立收束。

## 2. 产品目标

### 2.1 用户目标

完成本需求后，用户应能在一次正常播放中理解：

1. 飞书规则源需要由用户显式拉取；
2. 拉取、校验、规则集草稿、发布和候选生成是不同阶段；
3. 属性结果由多个有序来源共同结算；
4. 每个来源贡献正值、负值、Patch、无数值影响或边界处理中的哪一种；
5. 最终数字与完整 Trace、Issue 和版本证据一致；
6. 已发布 Snapshot 是冻结结果，上游变化只产生 UpgradeCandidate。

### 2.2 业务目标

- 让扎实但不易感知的计算、治理和追溯能力成为产品的显性价值；
- 缩短新用户理解生成链和属性结算逻辑的时间；
- 在演示、评审和日常使用中形成可识别的产品记忆点；
- 不牺牲高频用户效率，不要求用户每次等待动画结束。

### 2.3 成功指标

首轮上线以可观测行为和可用性验证为主，不虚构业务转化指标：

| 指标 | 目标 |
| --- | --- |
| 首次用户正确说出“显式拉取不等于发布” | 可用性测试 5 人中至少 4 人 |
| 首次用户正确指出最后一次属性变化的来源和 delta | 可用性测试 5 人中至少 4 人 |
| 正常属性结算总时长 | 2.25–2.45 秒，硬上限 2.5 秒 |
| 高频用户跳过后到达完整结果 | 一次操作，结果和证据无缺失 |
| 减少动态模式 | 无自动飞入、弹跳或 260ms 连续闪变 |
| 动效造成的领域结果差异 | 0；同一输入的结果、顺序和 hash 与无动效路径一致 |
| 目标设备流畅度 | 常规办公 Chromium 环境主动画无持续明显卡顿 |

## 3. 设计原则

### P1：动效解释因果，不制造因果

动效只消费服务端或领域层返回的阶段状态、稳定顺序、Trace、ValidationIssue、ActionAvailability 和冻结结果。前端不得根据颜色、标签、显示名称或动画完成状态推导规则结果。

### P2：快，但每一拍只有一个主角

正常播放保持高速。每一项遵循稳定焦点顺序：

```text
当前来源高亮
→ 来源卡进入结果区
→ delta 生效
→ 主数字更新
→ 解释与证据落位
```

解释区和证据可以延后出现，但不得改变真实计算顺序。

### P3：正向、负向、Patch、检查不是同一种奖励

不得用统一的“Combo”把所有步骤描述成连续奖励。统一使用中性的“链路 4/8”或“已结算 4/8”。正向、负向、Patch、`no_effect`、边界与舍入使用可辨识但克制的动作和文案；不能只靠颜色区分。

### P4：动画可以跳过，证据不能跳过

用户可以暂停、继续、直接看结果和重播。跳过只结束演出，不减少 Trace、Issue、版本、来源或最终证据。

### P5：成功感不能掩盖真实状态

只有权威状态确实成功时才展示完成重音。`BLOCKED`、`FAILED`、`WAITING_FOR_REVIEW`、`REMOTE_CHANGES_AVAILABLE`、`NON_FORMAL` 等状态必须使用自己的收束，不得播放“发布成功”或“已冻结”效果。

### P6：动效层不改变不可变边界

历史 ConfigurationSnapshot 不得因为重播、升级候选、规则变化或前端状态恢复而重新结算或改写。Snapshot 动效展示冻结事实；UpgradeCandidate 动效展示差异，不把候选表现为已生效。

## 4. 范围

### 4.1 本期范围

1. 飞书规则源显式拉取与规则集生成编排；
2. Model 属性来源高速结算；
3. 候选生成与自动物化的阶段反馈；
4. ValidationIssue、阻断、复核和失败恢复的动态反馈；
5. ConfigurationSnapshot 冻结与 UpgradeCandidate 差异呈现；
6. 全局暂停、直接看结果、重播和减少动态；
7. 可观测性、性能、键盘、缩放和屏幕阅读器验证。

### 4.2 不在本期

- 修改任何领域公式、Trace 顺序、Severity、Gate 或 ActionAvailability；
- 使用动画替代真实飞书连接、真实规则校验或真实发布；
- 为钩、漂、真饵和拟饵开启入口；
- 为被动技能执行或模拟运行逻辑；
- 引入积分、抽奖、概率闪光、宝箱或其他赌博式反馈；
- 为获得效果而生产装备插画、3D 模型或外部美术资源；
- 自动拉取飞书、自动发布 RuleSetVersion 或自动发布 Snapshot。

## 5. 核心体验一：飞书数据源与生成编排

### 5.1 初始态

- 固定展示唯一权威飞书工作簿及当前已知 revision；
- 初始状态使用“待连接”或“待拉取”，不得在用户操作前显示“同步中”；
- 主动作明确写为“从飞书拉取规则”；
- “拉取并生成候选”只能在产品确实提供组合动作且服务端返回对应 ActionAvailability 时使用，不能由前端把两个命令拼成一次假动作。

### 5.2 阶段序列

界面按真实状态展示：

```text
待拉取
→ 连接与读取工作簿
→ 创建 FeishuSourceRevision
→ 机器规则校验
→ 创建 RuleSet 草稿
→ 等待用户显式发布 RuleSetVersion
→ 派生模板与目标拉力匹配
→ SKU 组装
→ Model 候选与自动物化
→ 最终配置与发布检查
```

写回不等于拉取，拉取不等于发布。若真实链路停在 `REMOTE_CHANGES_AVAILABLE`、`WAITING_FOR_REVIEW` 或 `BLOCKED`，编排必须停在该阶段并展示原因与服务端动作。

### 5.3 动态表现

- 当前阶段是唯一主焦点；
- 已完成阶段快速沉淀为带 revision/hash 的证据节点；
- 后续阶段不得在真实状态到达前预演为成功；
- 规则来源、Method、Type、Function 必须分别留下轨迹，不能因视觉合并而合并领域层；
- 1.5kg 与 1.8kg 命中同一最近模板时，仍显示两个独立 SKU 节点，不用连续生长动画暗示插值。

### 5.4 中断与恢复

- 拉取失败保留已取得的非敏感诊断和重试入口；
- Revision 在运行中变化时，本次结果显示 `SUPERSEDED`，不得继续演成成功；
- 会话过期保留用户表单，重登后重新校验 revision；
- 重试复用原动作与幂等语义，不重复制造阶段证据。

## 6. 核心体验二：属性高速结算

### 6.1 数据契约

动效输入必须来自实际 `CalculationTraceEntry` 或等价权威 DTO，至少包含：

- 稳定 sequence；
- layer 和 sourceRef/sourceVersion；
- parameterKey、before、operation、operand、after、unit；
- effect 或 `no_effect`；
- warningIssueIds；
- inputHash/outputHash。

前端不得自行重排、重新计算 delta、合并 Technology 与成员 Affix 的重复贡献，或把缺失来源补成 `+0`。

### 6.2 展示模型

一个参数一次结算包含：

- 初始值；
- 来源总数；
- 当前来源名称、类型和作用；
- before → operation/operand → after；
- 当前链路进度；
- 最终值、边界、精度/舍入；
- 完整 Trace 与 Issue 入口。

Technology 只展示成员 Affix 的实际贡献。被动词条可以显示为“保存、计分、展示；不进入面板”，不得飞入主数字造成数值变化。

### 6.3 时序

推荐默认参数：

| 项目 | 要求 |
| --- | --- |
| 第一项 | 320–360ms，用于建立阅读模型 |
| 普通来源间隔 | 230–250ms |
| 负修正或 Patch | 280–320ms |
| 最终边界/舍入 | 360–420ms |
| 来源卡运动 | 440–480ms |
| 同屏来源卡 | 最多 2 张 |
| 撞击到数字更新 | 90–120ms |
| 解释更新 | 数字变化后 100–140ms |
| 证据落位 | 撞击后 140–180ms |
| 数字反馈 | 220–260ms |
| 最终结果锁定 | 220–280ms |

实现可以根据来源数量在已批准范围内压缩节拍，但必须保持稳定顺序和 2.5 秒上限。来源过多时不得无限延长；应显示代表性高速播放并保留完整证据，或根据产品验证后的阈值切换为分组结算。具体阈值在实现 Issue 中以真实数据验证后配置，不在本文硬编码。

### 6.4 最终收束

最后一项后进入独立完成阶段：

1. 主数字短促锁定；
2. 链路状态从“8/8”转为“结算完成”；
3. 显示最终边界与有效性；
4. 右侧切换为最终解释；
5. Trace 入口获得轻量提示。

若最终存在阻断，收束为“结算完成，发布被阻断”，明确显示 Issue；不能播放纯成功态。

## 7. 候选生成、发布和快照

### 7.1 候选生成

- 展示枚举总数、合法数、排除分组、截断、input hash 和耗时；
- 硬 deny 或缺 require 的候选进入排除，不得被高 Affinity 的视觉效果“救回”；
- 自动物化只对服务端已经确认的最高合法候选播放；
- 歧义对应、Revision 变化、超时或无合法候选分别有明确失败态。

### 7.2 Snapshot 冻结

冻结动效只在 SnapshotBuild 成功并取得稳定 snapshotId/hash 后触发。表现重点是“结果被封存且可复现”，不是庆祝数值变强。

必须展示：

- Snapshot ID、revision、RuleSetVersion；
- PatchSetHash、input/output hash；
- 冻结时间与发布人；
- 历史结果不会被上游变化改写。

失败 SnapshotBuild 不产生半冻结视觉状态。幂等重试成功后只生成一个完成证据。

### 7.3 UpgradeCandidate

上游变化时用差量流表现旧 Snapshot 与候选新结果的关系：

```text
旧 Snapshot（保持冻结）
→ UpgradeCandidate（可比较）
→ 人工批准
→ 显式发布新的 Snapshot
```

旧 Snapshot 始终留在画面或比较结构中，不得被候选覆盖、融化或替换。

## 8. 控制、可访问性与效率

### 8.1 用户控制

- 播放中提供暂停/继续；
- 提供“直接看结果”；
- 完成后提供重播；
- 重播只重放表现，不重复请求、写入、发布或创建 revision；
- 页面切换或恢复后以权威状态重建视图，不凭本地动画进度补写业务状态。

### 8.2 减少动态

当 `prefers-reduced-motion: reduce` 或产品内偏好启用时：

- 默认直接显示最终结果和完整证据；
- 不自动执行飞入、缩放、弹跳、速度线或连续闪变；
- 如需逐项理解，使用用户触发的“上一项/下一项”；
- 状态变化仍通过文字、图标和结构传达。

### 8.3 视觉与阅读

- 关键来源和 delta 不低于 11–12px；
- 证据条不低于 10–11px；
- 正、负、Patch、检查不得只靠颜色区分；
- 200% 和 400% 缩放下信息不重叠、不丢失控制；
- 键盘可以访问播放、暂停、跳过、重播、Trace 和 Issue；
- 屏幕阅读器不朗读每一帧数字变化；使用节制的 live region 汇报阶段或最终结果。

## 9. 技术策略与兼容边界

当前主技术栈为 React 19、TypeScript、Next/Vinext、Tailwind/CSS。第一阶段采用：

- React 显式状态机或 reducer 管理演出阶段；
- CSS transitions/keyframes 处理简单视觉过渡；
- Web Animations API 处理需要取消、跳过和精确时间控制的局部序列；
- 设计 token 管理时长、缓动、位移、强调强度和 reduced-motion；
- 现有 Lucide 图标与工作台色彩、间距和密度体系。

第一阶段不引入 GSAP、Framer Motion 或独立 Canvas/WebGL 渲染层。只有在原生方案无法满足取消、同步、性能或维护性验收，并有测量证据时，才由独立技术决策 Issue 评估依赖。

业务状态与演出状态必须分离：

```text
权威业务状态 / Trace
→ Motion Presentation Model
→ 可取消、可跳过的演出状态
```

演出完成不得反向提交业务成功。所有写动作仍由服务端 ActionAvailability、revision、hash、幂等和权限契约决定。

## 10. 验收场景

### 正常

- 用户显式拉取飞书，依次看到 revision、校验、RuleSet 草稿；只有显式发布后才进入已发布状态。
- 8 个来源在 2.5 秒内完成结算，用户能指出最后来源与 delta，最终 Trace 完整。
- SnapshotBuild 成功后才显示冻结完成，旧 Snapshot 保持不变。

### 边界

- `no_effect` 显示“本层无贡献”，不伪造 `+0`。
- 负修正、Patch、边界和舍入不累计为奖励 Combo。
- 大量来源按经验证的分组策略播放，但完整 Trace 不缺项。
- 1.5kg 与 1.8kg 可命中同模板，界面仍保留两个独立 SKU。

### 冲突

- 拉取期间 revision 变化，本次运行进入 `SUPERSEDED`，不演成成功。
- 硬 deny 与高 Affinity 同时存在，候选被排除并显示原因。
- Trace hash 不一致时停止成功收束并显示阻断。

### 恢复

- 暂停后从同一演出位置继续；
- “直接看结果”立即呈现完整最终证据；
- 重播不重复调用外部服务或创建业务记录；
- 页面恢复后根据权威业务状态重新构建表现。

### 权限与冻结

- 无对应 ActionAvailability 时，按钮显示禁用原因，动画不绕过动作；
- 历史 Snapshot 的重播、查看和比较不改变其 payload 或 hash；
- UpgradeCandidate 不覆盖旧 Snapshot。

## 11. 发布门槛

进入正式发布前必须具备：

1. 真实 Trace 驱动，而非原型假数据驱动；
2. 正常、边界、冲突、恢复、权限和历史冻结测试；
3. reduced-motion、键盘、200%/400% 缩放和屏幕阅读器验证；
4. 目标 Chromium 环境性能记录；
5. 跳过、重播不产生外部副作用的自动化证据；
6. 与无动效路径相比，领域结果、顺序、Issue、Snapshot 和 hash 完全一致。
